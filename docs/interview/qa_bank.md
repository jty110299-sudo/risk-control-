# Interview Q&A Bank

## 1. 项目整体设计是什么？

Risk Control Agent 是一个本地 MVP 风控监控工作台，覆盖数据治理、指标计算、规则预警、归因线索、报告生成、UI 展示和日志审计。

## 2. 为什么做风控 Agent？

风控监控需要把指标、规则、解释、报告和审计串起来。Agent 这个概念适合表达“辅助分析流程”，但项目不让 Agent 做最终业务决策。

## 3. 为什么用官方公开数据？

官方公开数据更适合 GitHub 作品集，来源可追溯，也能避免使用真实内部业务数据或个人金融数据。

## 4. 为什么不用完全 synthetic data？

完全 synthetic data 容易让项目脱离真实业务字段和数据限制。本项目只允许 synthetic data 作为明确标注的补充。

## 5. 为什么第一版用 Freddie Mac？

Freddie Mac Single-Family Loan-Level Dataset 和房贷表现、逾期迁移、信用风险监控更相关，适合作为 Credit Risk Monitoring 主线。

## 6. HMDA 的作用是什么？

HMDA 更适合做申请、审批、地区、机构和产品维度的辅助观察，不作为第一版贷后表现主数据源。

## 7. 为什么用规则引擎而不是 LLM 判断风险？

风控预警需要可解释、可审计、可校准。规则引擎可以绑定指标和阈值，LLM 更适合后续做报告辅助，不适合 MVP 中直接判断风险。

## 8. 什么是 bad_rate / delinquency_rate？

bad_rate 通常表示坏账或严重逾期比例，delinquency_rate 表示逾期比例。具体口径需要结合业务定义。

## 9. 为什么 Accuracy 不是风控核心指标？

风控样本通常类别不均衡，Accuracy 可能被多数类掩盖。AUC、KS、召回率、坏账捕获率、PSI 等更有参考价值。

## 10. AUC / KS / PSI 在项目中怎么规划？

当前项目只做 Model Monitoring 规划展示，未来可加入 AUC、KS、PSI、score distribution 和 feature drift。

## 11. 分群贡献怎么计算？

思路是比较不同 segment 的指标变化和样本占比，识别对整体风险变化贡献较大的群体。

## 12. 为什么贡献不是因果？

贡献分析只能说明某个分群与指标变化相关或贡献较高，不能证明该分群导致风险变化。

## 13. 报告如何避免胡说？

报告只基于 alert summary、root cause summary 和明确输入生成，缺少输入时不生成伪造报告，并区分事实、解释、建议和限制。

## 14. UI 为什么用 Streamlit？

Streamlit 适合 Python 数据分析项目，能快速展示指标、表格、JSON、Markdown 和本地流程按钮。

## 15. 为什么不做自动拒贷？

自动拒贷属于真实业务决策，需要严格审批、模型治理、合规和人工复核，不适合作品集 MVP。

## 16. 如何保证可审计？

每个脚本输出 run log，记录状态、时间、消息和输出路径；规则、配置和文档也保留来源和边界说明。

## 17. 数据怎么放 GitHub？

大型原始数据不提交，`data/raw` 和 `data/processed` 默认忽略，只保留 `.gitkeep`、模板和说明文档。

## 18. 项目当前限制是什么？

没有内置官方样本，没有 LLM，没有模型训练，没有生产数据库，没有调度和权限控制。

## 19. 后续怎么接入真实数据？

需要先明确数据权限、字段映射、脱敏策略和口径定义，再替换数据读取层，不改变安全边界。

## 20. 后续怎么接入 LLM？

LLM 可作为报告润色或问答辅助，但必须基于结构化指标和规则，不允许凭空判断风险。

## 21. 后续怎么做模型监控？

加入模型分数、标签和时间窗口后，可计算 AUC、KS、PSI、score drift、feature missing rate 等。

## 22. 怎么设计 alert rule？

规则应包含指标、阈值、比较方式、风险等级、触发条件、人工确认要求和业务解释。

## 23. 阈值怎么校准？

真实业务中需要结合历史样本、风险偏好、监管要求和策略收益成本进行校准。

## 24. 如何处理缺失数据？

缺失数据不补成假结果。脚本会跳过或给出 warnings，UI 显示空状态和下一步建议。

## 25. 如何处理没有输入数据？

完整 pipeline 可以运行，但前序脚本会 skipped 并写入日志，不生成伪造 outputs。

## 26. 为什么要 run log？

run log 让分析流程可追溯，便于知道输入是否存在、哪一步跳过、输出在哪里。

## 27. 这个项目和普通 dashboard 有什么区别？

它不仅展示图表，还包含数据治理、规则引擎、归因分析、报告生成、日志审计和安全边界。

## 28. 风控分析师怎么使用？

分析师可以查看风险总览、预警规则、贡献线索、报告和日志，并对高风险项做人工复核。

## 29. 项目最有技术含量的部分是什么？

模块化 pipeline、配置化规则、确定性报告、健康检查和安全脚本触发组合起来形成可复现流程。

## 30. 项目最有业务含量的部分是什么？

把 Credit Risk Monitoring、Risk Alert Rules、Human Confirmation、贡献非因果、数据治理和报告边界结合起来。
